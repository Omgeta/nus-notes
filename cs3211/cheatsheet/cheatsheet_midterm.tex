\section{Introduction}
Concurrency refers to a system performing multiple independent tasks in overlapping time periods.

Concurrency vs. Parallelism:
\begin{enumerate}[\roman*.]
  \item Concurrency: tasks progress in overlapping time 
  \item Parallelism: tasks also execute simultaneously 
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Enabled by multiple cores, each can have multiple hardware threads (SMT) 
      \item Hardware threads dictate parallel execution of kernel threads
    \end{itemize}
\end{enumerate}

Program Parallelization Steps:
\begin{enumerate}[\roman*.]
  \item Decomposition: split problem into parallel tasks
    \begin{itemize}[leftmargin=*]\vspace{2pt}
      \item Granularity: size of task
    \end{itemize}
  \item Scheduling: assign tasks to threads
  \item Orchestration: synchronisation and communication of tasks to satisfy dependencies
  \item Mapping: bind threads to physical cores 
\end{enumerate}

\subsection{Processes}
Process is an abstraction for a running program:
\begin{enumerate}[$-$]\vspace{-2pt}
  \item High system call overhead 
  \item OS allocates and initializes data structures 
  \item Communication costly through the OS
\end{enumerate}\vspace{-1pt}
{\centering
  \incimg[.8]{process}
\par}
\subsection{Threads}
Threads are independent execution flows within a process:
\begin{enumerate}[\roman*.]
  \item Shared: Resources, Address Space 
  \item Private: Thread ID, Registers, "Stack" (diff. SP)
\end{enumerate}\vspace{-1pt}
\begin{enumerate}[$+$]
  \item Efficient: cheaper creation and context switching
  \item Resource Sharing: process resources can be shared
\end{enumerate}\vspace{-1pt}
{\centering
  \incimg[0.7]{thread}
\par}
Implementations:
\begin{enumerate}[\roman*.]
  \item User-Threads are managed by a userspace library:
    \begin{enumerate}[$+$]\vspace{3pt}
    \item Fast: context switches have low overhead
  \end{enumerate}
  \begin{enumerate}[$-$]\vspace{5pt}
    \item Cannot parallelize to different cores 
    \item Threads blocking will block whole process
  \end{enumerate}
\item Kernel-Threads are implemented in the OS:
  \begin{enumerate}[$+$]\vspace{3pt}
    \item Parallel: threads can run across many cores 
  \end{enumerate}
  \begin{enumerate}[$-$]\vspace{5pt}
    \item Slower: thread operations are system calls
  \end{enumerate}
\end{enumerate}

\subsection{Synchronisation}

Race Condition:
\begin{enumerate}[\roman*.]
  \item Program outcome depends on timing/ordering of concurrent operations
\end{enumerate}

Data Races lead to undefined behaviour:
\begin{enumerate}[\roman*.]
  \item Two concurrent access to same memory location
  \item At least one access is a write
  \item No synchronization
\end{enumerate}

Mutual Exclusion synchronises access to shared resources:
\begin{enumerate}[\roman*.]
  \item Thread locks mutex to access shared resources 
  \item Same thread unlocks mutex once shared resources no longer used
\end{enumerate}

Symptoms of Incorrect Synchronization:
\begin{enumerate}[\roman*.]
  \item Deadlock: there is an execution where no progress is made when it should be possible 
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Indistinguishable from livelock 
    \end{itemize}
  \item Starvation: a thread cannot progress because other threads hold a required resource, which it does not get a chance to acquire
\end{enumerate}

\lstset{style=c}
Mechanisms:
\begin{enumerate}[\roman*.]
  \item Lock: primitive \lstinline|acquire()|, \lstinline|release()| 
  \item Semaphore: atomic $\geq 0$; \lstinline|acquire()|, \lstinline|release()|   
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Binary: represents single access 
      \item Counting: number of threads in CS $\leq N$
      \item Can be manipulated by different threads 
      \item No relation to data being controlled
    \end{itemize}
  \item Monitors: high-level implicit concurrency
  \item Messages: explicit model of communication and synchronisation based on atomic data transfer
\end{enumerate}

\subsection{Threads}

Operations:
\begin{enumerate}[\roman*.]
  \item Construct:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Function: \lstinline|std::thread t(f, args...)|
      \item Callable object: \lstinline|std::thread t{F{}, args...}|
      \item Lambda: \lstinline|std::thread t([]{ ... })|
    \end{itemize}
  \item Wait Once: \lstinline|t.join()|
    \begin{enumerate}[leftmargin=*, label=$\bullet$]\vspace{3pt}
      \item Check: \lstinline|t.joinable()|
      \item Must still join on exception paths 
    \end{enumerate}
  \item Detach: \lstinline|t.detach()|
    \begin{enumerate}[leftmargin=*, label=$-$]\vspace{3pt}
      \item Detached thread no longer joinable
      \item May outlive local variable references 
    \end{enumerate}
  \item Pass By Reference: \lstinline|std::ref(x)|
  \item Ownership: movable, not copyable
\end{enumerate}

\subsection{Mutexes}

Operations:
\begin{enumerate}[\roman*.]
  \item Construct: \lstinline|std::mutex m|
  \item Explicit: \lstinline|m.lock()|, \lstinline|m.unlock()|
  \begin{enumerate}[leftmargin=*, label=$-$]\vspace{3pt}
    \item Must unlock on every exit path, incl. exceptions
  \end{enumerate}
  \item RAII: \lstinline|std::lock_guard<std::mutex> g(m)| 
  \begin{enumerate}[leftmargin=*, label=$+$]\vspace{3pt}
    \item Locks on construction and unlocks on scope exit
  \end{enumerate}
  \item Flexible: \lstinline|std::unique_lock<std::mutex> lk(m)| 
  \begin{itemize}[leftmargin=*]\vspace{3pt}
    \item Adopt Existing Lock: \lstinline|lk(m, std::adopt_lock)|
    \item Defer Lock: \lstinline|lk(m, std::defer_lock)|
  \end{itemize}
  \begin{enumerate}[leftmargin=*, label=$+$]\vspace{5pt}
    \item Unlocks on scope exit if mutex owned
    \item Allows multiple lock/unlock within a scope 
    \item Needed for condition variables
    \item Moveable: can transfer lock ownership
  \end{enumerate}
  \item Multiple Mutexes (Deadlock-Free):
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Lock in total order, avoid nested locks
      \item Explicit: \lstinline|std::lock(m1, m2, ...)| 
      \item RAII: \lstinline|std::scoped_lock lk(m1, m2, ...)| 
    \end{itemize}
\end{enumerate}

\subsection{Condition Variables}

Condition variables sync actions without busy-waiting.

Operations:
\begin{enumerate}[\roman*.]
  \item {Construct: \lstinline|std::condition_variable cv|} 
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Works only on \lstinline|std::unique_lock<std::mutex>|
      \item Generic (any mutex-like, less efficient):\\\lstinline|std::condition_variable_any cva|
    \end{itemize}
  \item Wait: \lstinline|cv.wait(lk)| / \lstinline|cv.wait(lk, pred)| 
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Checks predicate with mutex locked
      \item If condition is met, returns
      \item If condition is unmet, unlocks mutex and blocks
    \end{itemize}
    \begin{enumerate}[leftmargin=*, label=$-$]\vspace{5pt}
      \item Spurious Wake: may wake without notification to check condition; so should be side effect-free
    \end{enumerate}
  \item Wake One Thread: \lstinline|cv.notify_one()|
  \item Wake All Threads: \lstinline|cv.notify_all()|
\end{enumerate}

\subsection{Monitors}
Monitors encapsulate shared data and operations enforced by a mutex and condition variables.
\begin{enumerate}[\roman*.]
  \item One thread executes monitor operations at a time 
  \item Threads wait for conditions and state changes
\end{enumerate}
Example:
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
class JobQueue {
  std::mutex m;
  std::condition_variable cv;
  std::queue<Job> q;
public:
  void enqueue(Job j) {
    {
      std::unique_lock<std::mutex> lk(m);
      q.push(j);
    }
    cv.notify_one();
  }

  Job dequeue() {
    std::unique_lock<std::mutex> lk(m);
    cv.wait(lk, [&]{ return !q.empty(); });
    Job j = q.front(); q.pop();
    return x;
  }
};
\end{lstlisting}

\subsection{Memory Model}
Memory model is the contract about the visible side effects of memory operations.
\begin{enumerate}[\roman*.]
  \item As-if: compiler/hardware may reorder operations if single-threaded observable behavior is unchanged.
\end{enumerate}

Ordering Relations:
\begin{enumerate}[\roman*.]
  \item Sequenced-Before: program order
  \item Synchronizes-With: suitably-tagged store $\to$ load on same atomic, if the load reads that value 
  \item Happens-Before: transitive closure of Sequenced-Before $\cup$ Synchronizes-With 
  \item Modification Order: per-atomic total order of writes consistent across threads 
  \item Coherence Order: per-atomic partial order of writes (totally ordered) and reads (unordered w.r.t others) consistent across threads
  \item Visible Side-Effect: write visible to a read 
\end{enumerate}

Atomics (\lstinline|std::atomic<T>|) have indivisible operations:
\begin{enumerate}[\roman*.]
  \item Load: \lstinline|a.load(std::memory_order)|
  \item Store: \lstinline|a.store(val, std::memory_order)| 
  \item RMW: \lstinline|a.fetch_add(val, std::memory_order)|, \lstinline|a.exchange(val, std::memory_order)|
  \item May be lock-free/implemented with mutex:\\
    \lstinline|a.is_lock_free()|, \lstinline|T::is_always_lock_free|
\end{enumerate}

Memory Models:
\begin{enumerate}[\roman*.]
  \item Sequentially Consistent (\lstinline|seq_cst|):
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item All threads agree on same order of operations
    \end{itemize}
  \item Relaxed (\lstinline|relaxed|):
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item No Synchronizes-With cross-thread guarantees
    \end{itemize}
  \item Acquire-Release (\lstinline|acquire|, \lstinline|release|, \lstinline|acq_rel|)
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Release store $\rightarrow$ Acquire load on same atomic can establish Synchronizes-With
      \item Writes before release become visible after the acquire (in the acquiring thread)
      \item Release Sequence: chain of RMWs on the same atomic after release; an acquire reading from the chain synchronizes-with the original release
    \end{itemize}
\end{enumerate}

\subsection{Lock-based Data Structures}
\lstset{style=cpp}
Lock-based structures protect shared state with mutexes.
\begin{enumerate}[\roman*.]
  \item Lock the right mutex before accessing protected data
  \item Hold locks for the minimum time
  \item Interface should not expose race conditions
  \item Deadlocks are possible
\end{enumerate}

\subsubsection{Stack (one global mutex)}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
void push(T x) {
  std::lock_guard<std::mutex> lk(m);
  data.push(std::move(x));
}

std::shared_ptr<T> pop() {
  std::lock_guard<std::mutex> lk(m);
  if (data.empty()) throw empty_stack();
  auto res = std::make_shared<T>(std::move(data.top()));
  data.pop();
  return res;
}
\end{lstlisting}\vspace{-1em}
\begin{enumerate}[$+$]
  \item Safe for concurrent calls, exception-safe
\end{enumerate}
\begin{enumerate}[$-$]\vspace{-2pt}
  \item Work is serialized
  \item No built-in waiting for items: must poll \lstinline|empty()| or catch \lstinline|empty_stack| exceptions
\end{enumerate}

\subsubsection{Queue (one mutex + notification)}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
void push(T x) {
  std::lock_guard<std::mutex> lk(mut);
  data_queue.push(std::move(x));
  data_cond.notify_one();
}

void wait_and_pop(T& value) {
  std::unique_lock<std::mutex> lk(mut);
  data_cond.wait(lk, [this]{ return !data_queue.empty(); });
  value = std::move(data_queue.front());
  data_queue.pop();
}
\end{lstlisting}\vspace{-1em}
\begin{enumerate}[$+$]
  \item \lstinline|wait_and_pop()| blocks until an item is available
  \item Safe for concurrent member calls
\end{enumerate}
\begin{enumerate}[$-$]\vspace{-2pt}
  \item One mutex still serializes queue operations
  \item \lstinline|notify_one()| + exception in \lstinline|wait_and_pop()| can leave other waiters asleep
\end{enumerate}
\colbreak
\subsubsection{Queue (push shared-pointers in the queue)}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
void push(T x) {
  auto data = std::make_shared<T>(std::move(x));
  std::lock_guard<std::mutex> lk(mut);
  data_queue.push(data);
  data_cond.notify_one();
}

std::shared_ptr<T> wait_and_pop() {
  std::unique_lock<std::mutex> lk(mut);
  data_cond.wait(lk, [this]{ return !data_queue.empty(); });
  auto res = data_queue.front();
  data_queue.pop();
  return res;
}
\end{lstlisting}\vspace{-1em}
\begin{enumerate}[$+$]
  \item Data can be allocated before locking
  \item Returned value can share ownership with the queue element
\end{enumerate}
\begin{enumerate}[$-$]\vspace{-2pt}
  \item One mutex still serializes queue operations
\end{enumerate}

\subsection{Queue (fine-grained; dummy node)}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
node* get_back() {
  std::lock_guard<std::mutex> lk(back_mutex);
  return back;
}

std::unique_ptr<node> pop_front() {
  std::lock_guard<std::mutex> lk(front_mutex);
  if (front.get() == get_back()) return nullptr;
  auto old_front = std::move(front);
  front = std::move(old_front->next);
  return old_front;
}

void push(T x) {
  auto new_data = std::make_shared<T>(std::move(x));
  auto p = std::make_unique<node>();
  node* new_back = p.get();
  std::lock_guard<std::mutex> lk(back_mutex);
  back->data = new_data;
  back->next = std::move(p);
  back = new_back;
}
\end{lstlisting}\vspace{-1em}
\begin{enumerate}[$+$]
  \item Separate \lstinline|front_mutex| and \lstinline|back_mutex|
  \item \lstinline|push()| only locks the back
  \item More concurrency than one global mutex
\end{enumerate}
\begin{enumerate}[$-$]\vspace{-2pt}
  \item Locking protocol and empty-check become more complex
\end{enumerate}
\vspace{-1em}
\colbreak
\subsubsection{Queue (more fine-grained)}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
std::unique_ptr<node> try_pop_head() {
  std::unique_lock<std::mutex> front_lk(front_mutex);
  if (front.get() == get_back()) return nullptr;

  std::unique_lock<std::mutex> node_lk(front->mutex);
  auto old_head = std::move(front);
  front = std::move(old_head->next);
  return old_head;
}

std::shared_ptr<T> try_pop() {
  std::unique_ptr<node> old_head;
  { old_head = try_pop_head(); }   // lock released before delete
  return old_head ? old_head->data : std::shared_ptr<T>();
}

void push(T x) {
  auto new_data = std::make_shared<T>(std::move(x));
  auto p = std::make_unique<node>();
  node* new_back = p.get();

  std::lock_guard<std::mutex> back_lk(back_mutex);
  std::lock_guard<std::mutex> node_lk(back->mutex);
  back->data = new_data;
  back->next = std::move(p);
  back = new_back;
}
\end{lstlisting}\vspace{-1em}
\begin{enumerate}[$+$]
  \item Per-node mutex
  \item More fine-grained synchronization between \lstinline|push()| and \lstinline|pop()|
\end{enumerate}
\begin{enumerate}[$-$]\vspace{-2pt}
  \item More complex locking protocol
  \item Must release locks before destroying the old node
\end{enumerate}

\subsection{Lock-Free Data Structures}
\lstset{style=cpp}

Lock-free structures avoid blocking library calls.
\begin{enumerate}[\roman*.]
  \item Obstruction-free: a thread completes if it eventually runs in isolation
  \item Lock-free: from any point, some thread completes
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Livelock-free but Starvation possible
    \end{itemize}
    \begin{enumerate}[leftmargin=*,label=$+$]\vspace{5pt}
      \item Maximum concurrency: some thread progresses
      \item Robust: if one thread dies, others can proceed
    \end{enumerate}
    \begin{enumerate}[leftmargin=*,label=$-$]\vspace{5pt}
      \item Atomics may be much slower than non-atomics
      \item Contention/write propagation hurt performance
      \item Cache ping-pong and false sharing
    \end{enumerate}
  \item Wait-free: every running thread completes regardless of execution speed of others
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Livelock- and Starvation-free; stronger lock-free
    \end{itemize}
    \begin{enumerate}[leftmargin=*,label=$+$]\vspace{5pt}
      \item Per-thread progress guaranteed
    \end{enumerate}
    \begin{enumerate}[leftmargin=*,label=$-$]\vspace{5pt}
      \item Harder to design and prove correct
      \item Often higher overhead than lock-free structures
    \end{enumerate}
\end{enumerate}

Risks:
\begin{enumerate}[\roman*.]
  \item Livelock: from unbounded restarts in response to other thread actions
  \item Starvation: loops with CAS, or algorithms with unbounded number of retries due to clashes
\end{enumerate}

Guidelines:
\begin{enumerate}[\roman*.]
  \item Start with \lstinline|std::memory_order_seq_cst|
  \item Use CAS loops carefully
  \item Watch for ABA
  \item Use safe memory reclamation
  \item Prefer lock-based unless lock-free progress needed
\end{enumerate}

\subsubsection{CAS Loop Pattern}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
int fetch_add(std::atomic<int>& value, int add) {
  int old_value = value.load();
  while (true) {
    int new_value = old_value + add;
    if (value.compare_exchange_weak(old_value, new_value))
      return old_value;
  }
}
\end{lstlisting}\vspace{-1em}

\subsubsection{Queue (initial idea)}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
void push(Job job) {
  Node* new_dummy = new Node();
  Node* work_node = m_queue_back.exchange(new_dummy);
  work_node->job = job;
  work_node->next.store(new_dummy);
}

std::optional<Job> try_pop() {
  Node* old_front = m_queue_front.load();
  while (true) {
    Node* new_front = old_front->next.load();
    if (new_front == QUEUE_END)
      return std::nullopt;
    if (m_queue_front.CAS(old_front, new_front))
      return old_front->job;
  }
}
\end{lstlisting}
\begin{enumerate}[$\bullet$]
  \item \lstinline|exchange()| gives each producer its exclusive dummy node
  \item \lstinline|try_pop()| advances the head with a CAS loop
\end{enumerate}

\subsubsection{Issue 1: Data races}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
Node* work_node = m_queue_back.exchange(new_dummy, stdmo::acq_rel);
work_node->next.store(new_dummy, stdmo::release);

Node* old_front = m_queue_front.load(stdmo::acquire);
Node* new_front = old_front->next.load(stdmo::acquire);
m_queue_front.compare_exchange_weak(old_front, new_front, stdmo::acq_rel);
\end{lstlisting}
\begin{enumerate}[$\bullet$]
  \item Producer vs.\ producer: \lstinline|m_queue_back.exchange(..., stdmo::acq_rel)|
  \item Producer vs.\ its consumer: release store to \lstinline|next|, acquire load from \lstinline|next|
  \item Producer vs.\ next consumer: acquire / acq\_rel accesses on \lstinline|m_queue_front|
\end{enumerate}

\subsubsection{Issue 2: ABA}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
struct alignas(16) GenNodePtr {
  Node* node;
  uintptr_t gen;
};

std::atomic<GenNodePtr> m_queue_front;
\end{lstlisting}\vspace{-1em}
\begin{enumerate}[$\bullet$]
  \item CAS may succeed even if a pointer changes away from A and back to A
  \item Fix: add a generation counter to the front pointer
\end{enumerate}

\subsubsection{Issue 3: Use-after-free}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
Node* get_recycled_node_or_allocate_new();
void add_node_to_recycling_stack(Node* node);
\end{lstlisting}
\begin{enumerate}[$\bullet$]
  \item Old front pointers may become stale
  \item Fix: do not delete nodes directly; reuse nodes via a lock-free recycling stack
\end{enumerate}

\subsubsection{Issue 4: Recycling race}
\begin{lstlisting}[basicstyle=\ttfamily\scriptsize]
Node* cur_stack_next =
  old_stack_top.node->next.load(stdmo::acquire);

node->next.store(old_stack_top.node, stdmo::release);
\end{lstlisting}
\begin{enumerate}[$\bullet$]
  \item Recycling also needs synchronization
  \item Use release / acquire on recycled-node \lstinline|next|
\end{enumerate}

\subsubsection{Issue 5: Serializability}
\begin{enumerate}[$\bullet$]
  \item The repaired queue can still be non-serializable: head / tail may be updated before fully linked
  \item Fix: Michael-Scott Queue prioritizes maintaining a valid linked list 
\end{enumerate}

\subsection{Producer-Consumer}
Processes share a bounded buffer of fixed size $K$, where producers add items until buffer full and consumers remove items when not empty.

Blocking Solution (init \lstinline|not_full = K, not_emp = 0|):\\
\begin{minipage}{0.45\columnwidth}
  \begin{lstlisting}
// Producer
x = produce();

wait(not_full);
wait(mutex);
buffer.add(x);
signal(mutex);
signal(not_emp);
  \end{lstlisting}
\end{minipage}
\begin{minipage}{0.5\columnwidth}
  \begin{lstlisting}
// Consumer
wait(not_emp);
wait(mutex);
x = buffer.get();
signal(mutex);
signal(not_full);

consume(item);
  \end{lstlisting}
\end{minipage}

\subsection{Reader-Writer}
Processes share a critical region, where readers can read simultaneously but writers must have exclusive access.
\vspace{-1em}
\begin{minipage}{0.45\columnwidth}
  \begin{lstlisting}
// Writer
wait(empty);
write();
signal(empty);
  \end{lstlisting}
\end{minipage}
\begin{minipage}{0.5\columnwidth}
  \begin{lstlisting}
// Reader 
wait(mutex);
readers++;
if (readers == 1)
  wait(empty);
signal(mutex);

read();

wait(mutex);
readers--;
if (readers == 0)
  signal(empty);
signal(mutex);
  \end{lstlisting}
\end{minipage}
\begin{enumerate}[$-$]
  \item Writer Starvation: possible if readers keep entering and \lstinline|empty| is never signalled. Solutions:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Queue: all pass through \lstinline|wait(queue)| 
      \item Writer-Priority: readers entering must \lstinline|wait(readTry)|; but first waiting writer blocks new readers with \lstinline|wait(readTry)| and does \lstinline|signal(readTry)| when done 
    \end{itemize}
\end{enumerate}

\subsection{Dining Philosophers}
$N$ philosophers around a circular table, with a single chopstick between. $2$ chopsticks are needed to eat.
\begin{minipage}{0.48\columnwidth}
  \begin{lstlisting}
// Philosopher
void p(int i) {
  while (1) {
    // think
    take(i);
    eat();
    put(i);
  }
}

void safe_to_eat (int i) {
  if (state[i] == HUNGRY && state[LEFT] != EATING && state[RIGHT] != EATING) {
    state[i] = EATING;
    signal(s[i]);
  }
}
  \end{lstlisting}
\end{minipage}
\begin{minipage}{0.45\columnwidth}
  \begin{lstlisting}
// Take Chopstick
void take(int i) {
  wait(mutex);
  state[i] = HUNGRY;
  safe_to_eat(i);
  signal(mutex);
  wait(s[i]);
}

// Put Chopstick
void put(int i) {
  wait(mutex);
  state[i] = THINKING;
  safe_to_eat(LEFT);
  safe_to_eat(RIGHT)
  signal(mutex);
}
  \end{lstlisting}
\end{minipage}
Limited Eater Solution, when at most $N-1$ philosophers eat concurrently, it's guaranteed at least one can eat:

void philosopher(int i) {

    wait(seats);

    wait(chopstick[RIGHT]);
    eat();
    signal(chopstick[RIGHT]);
    signal(chopstick[LEFT]);

\section{Testing}
\lstset{style=c}
Bug Types:
\begin{enumerate}[\roman*.]
\item Unwanted Blocking:
  \begin{itemize}[leftmargin=*]
    \item Deadlock/Livelock: no progress, operations are not "completed", effects do not become visible
    \item IO: waiting for external input
  \end{itemize}
\item Race Conditions:
  \begin{itemize}[leftmargin=*]\vspace{3pt}
    \item Data races $\Rightarrow$ undefined behavior (unsynchronized concurrent access)
    \item Broken invariants:
      \begin{itemize}[leftmargin=*]\vspace{5pt}
        \item Double-free 
        \item Dangling pointers $\implies$ UAF
        \item Memory corruption from read partial update
      \end{itemize}
    \item Lifetime issues: thread outlives data it accesses
  \end{itemize}
\end{enumerate}

Techniques:
\begin{enumerate}[\roman*.]
\item Review the code:
\begin{itemize}[leftmargin=*]\vspace{3pt}
\item Identify shared data and how it is protected
\item Track held mutexes and find lock ordering issues
\item Check cross-thread ordering enforcement
\item Validate lifetimes (e.g. \lstinline|join()| on exceptions)
\end{itemize}
\item Testing:
\begin{itemize}[leftmargin=*]\vspace{3pt}
\item Run the smallest amount of code that can demonstrate the problem
\item Remove concurrency to confirm it is concurrency-related
\item Run on multicore and (when useful) single-core
\end{itemize}
\end{enumerate}

\subsection{Dynamic Instrumentation}

Valgrind:
\begin{enumerate}[\roman*.]
  \item Shadows registers and memory to validate memory operations:\\each allocation freed once, each access to allocated space, each reach from initialized memory
  \item High overhead: \lstinline|10-20x|
  \item Usage: \lstinline|valgrind --tool=memcheck <prog ...>|
\end{enumerate}

Helgrind:
\begin{enumerate}[\roman*.]
  \item Intercepts \lstinline|pthread| operations to validate thread operations:\\miuse of POSIX threads API, potential deadlocks from lock-order cycles, data races from happens-before analysis
  \item Very high overhead: \lstinline|~100x|
  \item Usage: \lstinline|valgrind --tool=helgrind <prog ...>|
\end{enumerate}

\subsection{Compiler Instrumentation}

Sanitizers:
\begin{enumerate}[\roman*.]
  \item Compile-time instrumentation; supported by \lstinline|clang| and \lstinline|gcc|
  \item Lower overhead: \lstinline|~5-10x|
  \item Instruments memory accesses / function boundaries and tracks memory state
\end{enumerate}

AddressSanitizer (ASan):
\begin{enumerate}[\roman*.]
  \item Detects addressability issues
  \item Typical cost: \lstinline|~2x| slowdown, \lstinline|1.5-3x| memory overhead
  \item Usage: \lstinline|-fsanitize=address|
\end{enumerate}

ThreadSanitizer (TSan):
\begin{enumerate}[\roman*.]
  \item Detects data races using happens-before tracking
  \item Also reports thread / mutex / lock-order issues
  \item Typical cost: \lstinline|5-15x| slowdown, \lstinline|5-10x| memory overhead
  \item Usage: \lstinline|-fsanitize=thread -fPIE -pie|
\end{enumerate}

\subsection{Classes}

Class Skeleton:
\begin{lstlisting}
class C {
  std::string x;
public:
  C(): x("default") {
    // default constructor
  }

  C(int i): x(std::string(i)) {
    // custom constructor
  }

  C(const C& other) : x(other.x) {
    // copy constructor
  }

  C& operator=(const C& other) {
    // copy assignment operator
    x = other.x;
    return *this;
  }

  C(C&& other): x(std::move(other.x)) {
    // move constructor
  }

  C& operator=(C&& other) {
    // move assignment operator
    x = std::move(other.x);
    return *this;
  }

  ~C() {
    // destructor
  }
};
\end{lstlisting}

Rule of 3: if a class needs a custom destructor, copy constructor, or copy assignment operator then it usually needs all 3

\subsection{Smart Pointers}

Smart pointers manage heap resources automatically.

Ownership:
\begin{enumerate}[\roman*.]
  \item \lstinline|std::unique_ptr<T>|: exclusive ownership
  \item \lstinline|std::shared_ptr<T>|: shared ownership via atomic reference counting
  \item \lstinline|std::weak_ptr<T>|: non-owning observer of a \lstinline|shared_ptr|
\end{enumerate}

Basic Methods:
\begin{enumerate}[\roman*.]
  \item Construct:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item \lstinline|auto p = std::make_unique<T>(args...)|
      \item \lstinline|auto p = std::make_shared<T>(args...)|
    \end{itemize}
  \item Access:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item Dereference: \lstinline|*p|, \lstinline|p->member|
      \item Raw pointer: \lstinline|p.get()|
    \end{itemize}
  \item Reset / replace:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item \lstinline|p.reset()|
      \item \lstinline|p.reset(new T(...))|
    \end{itemize}
  \item Move ownership:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item \lstinline|std::unique_ptr| is movable, not copyable
      \item \lstinline|auto q = std::move(p)|
    \end{itemize}
  \item Shared count:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item \lstinline|p.use_count()|
      \item Copying a \lstinline|shared_ptr| increments the count
    \end{itemize}
  \item Weak access:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item \lstinline|w.lock()| gives a \lstinline|shared_ptr| if object still exists
      \item \lstinline|w.expired()| checks whether object has already been destroyed
    \end{itemize}
  \item Unique-pointer release:
    \begin{itemize}[leftmargin=*]\vspace{3pt}
      \item \lstinline|p.release()| returns raw pointer and gives up ownership
    \end{itemize}
\end{enumerate}
